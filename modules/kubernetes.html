Module 09 — Orchestration

Kubernetes — Container Orchestration

What is Kubernetes?

Kubernetes (K8s) is an open-source system that automatically manages containerised applications at scale. It handles starting, stopping, distributing, scaling, and healing containers — so you don't have to do it manually.

🍽️ Analogy — Restaurant Manager

You have 50 waiters (containers) serving 500 customers across 5 floors (servers). The restaurant manager (Kubernetes): decides which floor each waiter goes to, replaces waiters who call in sick automatically, adds more waiters when the restaurant gets busy, and ensures every customer gets served. You just tell the manager "I want 50 waiters at all times" — the rest is handled automatically.

Kubernetes Architecture — Deep Dive

A Kubernetes cluster has two parts: the Control Plane (the brain) and Worker Nodes (the muscle). Here's what each component actually does:

🚪
kube-apiserver
The front door to the cluster. Every command you run with kubectl goes through here. Validates and processes all API requests.
🗄️
etcd
Distributed key-value database — the single source of truth for the entire cluster state. If etcd is lost, the cluster is lost.
📅
kube-scheduler
Watches for new Pods with no node assigned and picks the best node to run them on — based on resource availability, affinity rules, and constraints.
🔄
controller-manager
Runs control loops that watch the actual cluster state and reconcile it toward the desired state. Runs the Deployment, ReplicaSet, and Node controllers.
🤖
kubelet
Agent running on every worker node. Receives Pod specs from the API server and makes sure the right containers are running on that node.
🌐
kube-proxy
Maintains network rules on each node. Handles routing traffic to the right Pod when you access a Service.

Core Kubernetes Objects — Explained

Pod — The Smallest Unit

A Pod wraps one or more containers that share the same network and storage. Pods are ephemeral — they can be killed and replaced at any time. Never rely on a Pod's IP address directly.

pod.yaml
apiVersion: v1 kind: Pod metadata: name: my-app-pod labels: app: my-app spec: containers: - name: my-app image: nginx:latest ports: - containerPort: 80 resources: requests: # minimum guaranteed resources cpu: "100m" # 100 millicores = 0.1 CPU memory: "128Mi" limits: # maximum allowed resources cpu: "500m" memory: "256Mi"

ReplicaSet — Maintain Pod Count

A ReplicaSet ensures a specified number of identical Pods are always running. If a Pod dies, ReplicaSet starts a replacement. In practice you rarely create ReplicaSets directly — Deployments manage them for you.

Deployment — Desired State Management

The most common object you'll work with. A Deployment manages ReplicaSets and adds rolling updates and rollbacks on top.

deployment.yaml
apiVersion: apps/v1 kind: Deployment metadata: name: my-app namespace: production spec: replicas: 3 selector: matchLabels: app: my-app strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 # max extra pods during update maxUnavailable: 0 # zero downtime template: metadata: labels: app: my-app spec: containers: - name: my-app image: yourusername/my-app:2.0 ports: - containerPort: 3000 resources: requests: cpu: "100m" memory: "128Mi" limits: cpu: "500m" memory: "512Mi" livenessProbe: # restart if this fails httpGet: path: /health port: 3000 initialDelaySeconds: 15 periodSeconds: 20 readinessProbe: # only send traffic if this passes httpGet: path: /ready port: 3000 initialDelaySeconds: 5 periodSeconds: 10
💡 Liveness vs Readiness Probes

Liveness probe: "Is this container still alive?" If it fails, Kubernetes kills and restarts the container. Use for detecting deadlocks or crashes.

Readiness probe: "Is this container ready to receive traffic?" If it fails, Kubernetes removes the Pod from the Service's load balancer — but doesn't restart it. Use for app warmup, database connection checks.

StatefulSet — For Stateful Apps

Like a Deployment, but for apps that need stable identity — databases, message queues. Each Pod gets a persistent name (db-0, db-1, db-2) and its own persistent volume that follows it even when rescheduled.

DaemonSet — One Pod Per Node

Ensures exactly one copy of a Pod runs on every node in the cluster. Used for cluster-wide services: log collectors (Fluentd), monitoring agents (Prometheus node-exporter), networking plugins.

Job & CronJob — Run to Completion

A Job runs one or more Pods until they complete successfully — database migrations, batch processing. A CronJob runs a Job on a schedule (cron syntax) — nightly reports, scheduled cleanups.

cronjob.yaml
apiVersion: batch/v1 kind: CronJob metadata: name: db-backup spec: schedule: "0 2 * * *" # every day at 2am jobTemplate: spec: template: spec: containers: - name: backup image: postgres:15 command: ["pg_dump", "-h", "db", "myapp"] restartPolicy: OnFailure

Services — Stable Networking for Pods

Pods come and go — their IP addresses change every time. A Service provides a stable endpoint that always routes to the right Pods, regardless of which Pods are currently running.

🔒
ClusterIP (default)
Internal-only. Accessible only within the cluster. Use for communication between microservices — e.g. web app talking to a database service.
🔌
NodePort
Opens a port (30000–32767) on every node's IP. Accessible from outside the cluster. Simple but not suitable for production load balancing.
⚖️
LoadBalancer
Provisions a cloud load balancer (AWS ELB, GCP LB). The standard way to expose apps publicly in cloud environments.
🔗
ExternalName
Maps a Service to an external DNS name. Use to give an internal alias to an external service like a managed database.
service.yaml — All Types
# ClusterIP — internal only apiVersion: v1 kind: Service metadata: name: my-app-internal spec: type: ClusterIP selector: app: my-app ports: - port: 80 # port other pods use to reach this service targetPort: 3000 # port your container listens on --- # LoadBalancer — public internet access apiVersion: v1 kind: Service metadata: name: my-app-public spec: type: LoadBalancer selector: app: my-app ports: - port: 80 targetPort: 3000

ConfigMaps & Secrets

Never hardcode configuration or credentials inside your container image. Kubernetes provides two objects to inject this at runtime.

configmap-and-secret.yaml
# ConfigMap — non-sensitive config apiVersion: v1 kind: ConfigMap metadata: name: app-config data: APP_ENV: production LOG_LEVEL: info MAX_CONNECTIONS: "100" --- # Secret — sensitive data (base64 encoded) apiVersion: v1 kind: Secret metadata: name: app-secrets type: Opaque data: DB_PASSWORD: c2VjcmV0cGFzcw== # base64 of "secretpass" API_KEY: bXlhcGlrZXkxMjM=
deployment.yaml — Using ConfigMap & Secret
spec: containers: - name: my-app image: yourusername/my-app:1.0 envFrom: - configMapRef: name: app-config # inject all ConfigMap keys as env vars - secretRef: name: app-secrets # inject all Secret keys as env vars
Terminal — Create a Secret
# Create secret from literal values (avoids checking plaintext into git) kubectl create secret generic app-secrets \ --from-literal=DB_PASSWORD=secretpass \ --from-literal=API_KEY=myapikey123 # View secrets (values are base64 encoded) kubectl get secret app-secrets -o yaml

Namespaces — Logical Isolation

Namespaces let you divide a single cluster into virtual clusters. Use them to separate environments (dev, staging, production) or teams within the same physical cluster.

Terminal — Namespaces
# Create namespaces kubectl create namespace development kubectl create namespace production # Deploy to a specific namespace kubectl apply -f deployment.yaml -n production # List pods in a namespace kubectl get pods -n production # List pods across ALL namespaces kubectl get pods --all-namespaces # Set a default namespace for your session kubectl config set-context --current --namespace=production

Persistent Volumes — Storage in K8s

Like Docker volumes but cluster-wide. A PersistentVolume (PV) is actual storage (AWS EBS, NFS, local disk). A PersistentVolumeClaim (PVC) is a request for that storage by a Pod.

Pod → requests storage via → PVC → bound to → PV → backed by → AWS EBS / NFS / etc
pvc.yaml
apiVersion: v1 kind: PersistentVolumeClaim metadata: name: db-storage spec: accessModes: - ReadWriteOnce # one node can read/write at a time resources: requests: storage: 10Gi # request 10GB storageClassName: gp2 # AWS EBS SSD

Essential kubectl Commands

Terminal — kubectl Cheatsheet

Helm — The Package Manager for Kubernetes

Deploying an app on Kubernetes means writing many YAML files — Deployment, Service, ConfigMap, Secret, Ingress, PVC. Helm bundles all of this into a Chart — a reusable, versioned package you can install with one command.

📦 Analogy — apt / npm for Kubernetes

Just like apt install nginx installs Nginx with all its configuration on Ubuntu, helm install my-nginx nginx/nginx installs Nginx on Kubernetes with all its Deployments, Services, and config — no manual YAML writing needed.

Terminal — Helm Basics
# Install Helm curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash # Add a chart repository helm repo add stable https://charts.helm.sh/stable helm repo update # Search for a chart helm search repo nginx # Install a chart helm install my-nginx nginx/nginx-ingress # List installed releases helm list # Upgrade a release helm upgrade my-nginx nginx/nginx-ingress --set replicaCount=3 # Rollback to previous version helm rollback my-nginx 1 # Uninstall helm uninstall my-nginx

Self-Healing & Auto-Scaling

🔄 Self-Healing in Action

You told Kubernetes: "I want 3 replicas." If one Pod crashes at 3am, Kubernetes automatically starts a replacement within seconds — without anyone waking up. This is called self-healing and it works because the controller-manager constantly compares desired state (3 replicas) vs actual state (2 running) and acts to fix the difference.

Horizontal Pod Autoscaler (HPA) automatically scales your Deployment up or down based on CPU or memory usage:

hpa.yaml
apiVersion: autoscaling/v2 kind: HorizontalPodAutoscaler metadata: name: my-app-hpa spec: scaleTargetRef: apiVersion: apps/v1 kind: Deployment name: my-app minReplicas: 2 maxReplicas: 20 metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 70 # scale up if CPU > 70%

🚀 Mini Project — Deploy a Full App on K8s

Use Minikube (free local Kubernetes) to deploy a complete application:

  1. Install Minikube: minikube start
  2. Create a Namespace: kubectl create namespace myapp
  3. Create a ConfigMap for app config and a Secret for DB password
  4. Write a Deployment with 3 replicas, resource limits, liveness and readiness probes
  5. Expose it with a Service (NodePort for local testing)
  6. Add a HPA to auto-scale between 2 and 10 replicas at 70% CPU
  7. Delete a Pod manually — watch Kubernetes recreate it instantly
  8. Roll out a new image version — observe the rolling update with kubectl rollout status
  9. Rollback with kubectl rollout undo

Bonus: Install a Helm chart (e.g. Prometheus) with a single command and explore what it created.

✅ Module Summary
  • Control Plane (API server, etcd, scheduler, controller-manager) manages cluster state
  • Worker Nodes run Pods via kubelet and handle networking via kube-proxy
  • Pod = containers, Deployment = desired state + rolling updates, ReplicaSet = pod count
  • StatefulSet for databases, DaemonSet for node-level agents, CronJob for scheduled tasks
  • Services provide stable endpoints: ClusterIP (internal), LoadBalancer (public)
  • ConfigMaps for config, Secrets for credentials — never hardcode in images
  • Liveness probes restart unhealthy containers; Readiness probes control traffic routing
  • HPA automatically scales pods based on CPU/memory — always set resource limits
  • Helm packages all K8s YAML into reusable, versioned charts
