Module 08 — Containers

Docker — Containerization

The "Works on My Machine" Problem

Developer A writes an app on Windows with Python 3.9. Developer B pulls the code on Mac with Python 3.11. The production server runs Linux with Python 3.7. Everyone is confused when things break at each step. This is one of the most frustrating problems in software engineering.

🚢 Analogy — Shipping Containers

Before shipping containers, loading cargo onto ships was chaotic. Different sizes, different methods, things breaking. Then someone invented the standard shipping container — one size, works on any ship, any truck, any train, any port in the world.

Docker containers are the shipping containers of software. You pack your app, its dependencies, its runtime — everything — into a single container. It runs identically on your laptop, your colleague's laptop, Jenkins, and production. No surprises.

How Docker Actually Works Under the Hood

Docker isn't magic. It uses two Linux kernel features to isolate containers from each other and from the host:

🔒
Namespaces
Each container gets its own view of the system — its own process list, network, filesystem, and hostname. Containers can't see each other's processes.
📊
cgroups
Control Groups limit how much CPU, RAM, and disk a container can use. Prevents one container from starving the rest.
📁
Union File System
Docker images are built in layers. Each Dockerfile instruction adds a layer. Layers are shared and cached — making builds fast and storage efficient.
🔄
Container Runtime
Docker uses containerd under the hood to actually run containers. The Docker CLI is just a friendly interface on top.

Core Docker Concepts

📄
Dockerfile
A recipe/blueprint that describes how to build your container image step by step.
💿
Image
A snapshot built from a Dockerfile. Read-only, layered, and shareable. Like a frozen template.
📦
Container
A running instance of an image. Isolated, lightweight, and disposable.
🗄️
Registry
Docker Hub — a store of images. Pull official images or push your own.

Writing a Dockerfile — Node.js App

Dockerfile
# Start from an official Node.js base image FROM node:18-alpine # Set working directory inside the container WORKDIR /app # Copy dependency files FIRST (layer cache trick) COPY package*.json ./ # Install dependencies RUN npm install # Copy the rest of the application code COPY . . # Expose port 3000 EXPOSE 3000 # Command to run when container starts CMD ["node", "app.js"]
💡 Why COPY package.json before COPY . ?

Docker caches each layer. If you copy package.json first and run npm install, Docker only re-runs npm install when package.json changes — not every time your code changes. Always order Dockerfile instructions from least changing to most changing for fast rebuilds.

Multi-Stage Builds — Smaller, Safer Images

By default, your production image contains everything used to build the app — compilers, dev tools, test frameworks. You don't need any of that in production. It makes the image bigger and adds security risk.

🏭 Analogy — Factory vs Delivery Truck

A factory needs heavy machinery and raw materials to build a product. But the delivery truck only carries the finished product — not the factory. Multi-stage builds work the same way: use a full build environment in Stage 1, copy only the compiled output into a minimal final image in Stage 2.

Dockerfile — Multi-Stage (Java App)
# ── Stage 1: BUILD ────────────────────────────── # Full JDK + Maven to compile the app FROM maven:3.9-eclipse-temurin-17 AS builder WORKDIR /app COPY pom.xml . RUN mvn dependency:go-offline COPY src ./src RUN mvn package -DskipTests # ── Stage 2: PRODUCTION ───────────────────────── # Tiny JRE-only image — no compiler, no Maven, no source code FROM eclipse-temurin:17-jre-alpine WORKDIR /app # Copy ONLY the compiled JAR from the builder stage COPY --from=builder /app/target/myapp.jar app.jar EXPOSE 8080 CMD ["java", "-jar", "app.jar"]
Terminal — Size Comparison
docker images REPOSITORY TAG SIZE myapp single-stage 612MB ← JDK + Maven + source code myapp multi-stage 98MB ← just JRE + compiled JAR
✅ When to use Multi-Stage Builds

Compiled languages — Java, Go, Rust, C++ — benefit most. Node.js apps also benefit if you have a TypeScript or webpack build step. If your app has no build step, a single stage is fine.

Docker Networking

Containers are isolated by default — they can't talk to each other unless you connect them to the same network.

🌉
Bridge (default)
Containers on the same bridge network communicate by container name. Isolated from the host. Docker Compose uses this automatically.
🖥️
Host
Container shares the host's network stack directly. No port mapping needed. Use for performance-critical apps.
🚫
None
No network at all. Completely isolated. Used for batch jobs that don't need internet access.
🔗
Custom Bridge
Your own named network with automatic DNS. Containers resolve each other by service name. Best practice for multi-container apps.
Terminal — Docker Networking
# Create a custom network docker network create my-network # Run containers on the same network docker run -d --name app --network my-network my-node-app:1.0 docker run -d --name db --network my-network postgres:15 # 'app' can now reach 'db' by name — Docker handles DNS # Connection string: postgresql://db:5432/myapp # List all networks docker network ls NETWORK ID NAME DRIVER a1b2c3d4e5f6 bridge bridge b2c3d4e5f6a1 my-network bridge c3d4e5f6a1b2 host host

Docker Volumes — Persistent Data

Containers are ephemeral — when removed, all data inside is gone. If your database container is deleted, all data disappears with it. Volumes solve this by storing data outside the container lifecycle.

📂
Named Volumes
Docker manages the location on disk. Best for databases and persistent app data. Survives container removal.
🔗
Bind Mounts
Map a specific folder from your host machine into the container. Perfect for local development — code changes reflect instantly.
⚡
tmpfs Mounts
Stored in memory only. Fast but not persistent. Use for sensitive temporary data you don't want written to disk.
Terminal — Volumes
# Named volume — data persists after container is removed docker run -d \ --name postgres-db \ -v db-data:/var/lib/postgresql/data \ -e POSTGRES_PASSWORD=secret \ postgres:15 # Bind mount — code changes reflect live (dev mode) docker run -d \ --name dev-app \ -v $(pwd)/src:/app/src \ -p 3000:3000 \ my-node-app:1.0 # List all volumes docker volume ls DRIVER VOLUME NAME local db-data # See where Docker stored it on disk docker volume inspect db-data

Docker Compose — Deep Dive

Docker Compose defines your entire stack — app, database, cache — in one YAML file. One command starts everything. One command stops it.

docker-compose.yml — Production-Style
version: '3.8' services: # ── Web Application ───────────────────────── web: build: . ports: - "3000:3000" environment: - NODE_ENV=production - DB_HOST=db - REDIS_HOST=cache env_file: - .env # load secrets from file depends_on: db: condition: service_healthy # wait for DB ready signal restart: unless-stopped networks: [app-network] # ── PostgreSQL Database ────────────────────── db: image: postgres:15-alpine env_file: .env volumes: - db-data:/var/lib/postgresql/data healthcheck: test: ["CMD-SHELL", "pg_isready -U postgres"] interval: 10s timeout: 5s retries: 5 networks: [app-network] # ── Redis Cache ────────────────────────────── cache: image: redis:7-alpine volumes: - redis-data:/data networks: [app-network] volumes: db-data: redis-data: networks: app-network: driver: bridge
💡 depends_on vs healthcheck

depends_on alone only waits for a container to start — not the service inside to be ready. Postgres can be "started" while the database is still initialising. Add a healthcheck with condition: service_healthy to wait until it actually accepts connections.

Terminal — Compose Commands
docker compose up -d # start everything in background docker compose logs -f # tail logs from all services docker compose logs -f web # logs from one service docker compose up -d --build web # rebuild and restart one service docker compose exec db psql -U postgres # shell into db docker compose down # stop, keep volumes docker compose down -v # stop AND wipe volumes

Image Best Practices & Security

Dockerfile — Best Practices
# ✅ Use specific slim base images FROM node:18-alpine # ~5MB vs ~900MB for full debian # ✅ Combine RUN commands to reduce layers RUN apt-get update && \ apt-get install -y curl && \ rm -rf /var/lib/apt/lists/* # clean cache in SAME layer # ✅ Run as non-root user — never run as root in prod RUN addgroup -S appgroup && adduser -S appuser -G appgroup USER appuser
.dockerignore
node_modules .git .env *.log README.md tests/ docs/
⚠️ Never put secrets in a Dockerfile

Don't write ENV DB_PASSWORD=mysecret in a Dockerfile — it bakes the secret into every image layer permanently. Anyone with the image can read it with docker history. Use --env-file at runtime, Docker secrets, or AWS Secrets Manager instead.

🚀 Mini Project — Full Stack App with Docker Compose

Build a complete 3-tier app:

  1. Write a Node.js/Express API that reads and writes to PostgreSQL
  2. Write a multi-stage Dockerfile for the app
  3. Write a docker-compose.yml with: web, db (with healthcheck), and redis cache
  4. Use a .env file for secrets, load via env_file
  5. Add a named volume for Postgres data
  6. Scan the final image with Trivy
  7. Run docker compose down && docker compose up -d — verify DB data persists
✅ Module Summary
  • Docker uses namespaces + cgroups for isolation — not full VMs
  • Order Dockerfile from least to most changing for fast layer caching
  • Multi-stage builds strip build tools from production images — huge size savings
  • Use custom bridge networks so containers talk by name
  • Named volumes keep database data alive after containers are removed
  • Docker Compose manages multi-container stacks — always use healthchecks
  • Use alpine images, .dockerignore, non-root users, and Trivy scanning in prod
