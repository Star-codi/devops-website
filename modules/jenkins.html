Module 07 — CI/CD

Jenkins — Continuous Integration & Delivery

What is CI/CD?

CI (Continuous Integration) means every time a developer pushes code, it's automatically built and tested. Bugs are caught immediately — not weeks later.

CD (Continuous Delivery) means if all tests pass, the code is automatically packaged and made ready to deploy. Continuous Deployment goes one step further — it deploys straight to production with no human intervention.

🏭 Analogy — Factory Assembly Line

Old way: workers build the entire car, quality check it at the end — finding defects when it's expensive to fix. CI/CD way: every step of the assembly line automatically checks quality. Problems are caught immediately at the source. Jenkins is the factory manager — it triggers each step automatically the moment a developer pushes code.

Jenkins Architecture

Jenkins uses a Controller + Agent model. The Controller manages pipelines and the UI. Agents (workers) actually run the pipeline jobs — on separate machines, Docker containers, or cloud instances.

┌─────────────────────────────────────────────────────────┐ │ JENKINS CONTROLLER │ │ - Web UI (port 8080) │ │ - Pipeline scheduling & management │ │ - Plugin management │ │ - Stores build history & logs │ └──────────┬──────────────┬────────────────┬──────────────┘ │ │ │ ▼ ▼ ▼ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Agent 1 │ │ Agent 2 │ │ Agent 3 │ │ (Linux VM) │ │ (Docker) │ │ (Windows) │ │ Java builds │ │ Node builds │ │ .NET builds │ └──────────────┘ └──────────────┘ └──────────────┘
💡 Why Agents?

Running all builds on the Controller is a bad practice — it becomes a bottleneck and a single point of failure. Agents let you run many builds in parallel, on different environments, without touching the Controller.

Jenkinsfile — Declarative Pipeline

A Jenkinsfile defines your pipeline as code. It lives in your repo root alongside your application code — versioned, reviewable, and auditable.

There are two syntax styles: Declarative (structured, recommended for beginners) and Scripted (Groovy-based, more flexible). We'll use Declarative.

Jenkinsfile — Basic Structure
pipeline { agent any // run on any available agent environment { // pipeline-wide environment variables APP_NAME = 'my-app' DOCKER_HUB = 'yourusername' } stages { stage('Checkout') { steps { git branch: 'main', url: 'https://github.com/yourusername/my-app.git' } } stage('Build') { steps { sh 'mvn clean package -DskipTests' } } stage('Test') { steps { sh 'mvn test' } post { always { // publish test results even if tests fail junit 'target/surefire-reports/*.xml' } } } stage('Docker Build & Push') { steps { // use credentials stored in Jenkins — never hardcode withCredentials([usernamePassword( credentialsId: 'dockerhub-creds', usernameVariable: 'DOCKER_USER', passwordVariable: 'DOCKER_PASS' )]) { sh """ docker build -t ${DOCKER_HUB}/${APP_NAME}:${BUILD_NUMBER} . echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin docker push ${DOCKER_HUB}/${APP_NAME}:${BUILD_NUMBER} """ } } } stage('Deploy to Kubernetes') { steps { sh """ kubectl set image deployment/${APP_NAME} \ ${APP_NAME}=${DOCKER_HUB}/${APP_NAME}:${BUILD_NUMBER} kubectl rollout status deployment/${APP_NAME} """ } } } post { success { echo '✅ Pipeline succeeded! App deployed.' } failure { echo '❌ Pipeline failed — check the logs above.' // emailext to: 'team@company.com', subject: 'Build Failed' } always { // clean up workspace after every run cleanWs() } } }

Advanced Pipeline Features

Parallel Stages — Speed Up Your Pipeline

Run independent stages simultaneously instead of one at a time. Unit tests and security scans don't depend on each other — run them in parallel and cut pipeline time in half.

Jenkinsfile — Parallel Stages
stage('Quality Checks') { parallel { stage('Unit Tests') { steps { sh 'mvn test' } } stage('Code Coverage') { steps { sh 'mvn jacoco:report' } } stage('Security Scan') { steps { sh 'trivy image my-app:latest' } } } }

Parameters — Make Pipelines Interactive

Jenkinsfile — Parameters
pipeline { agent any parameters { choice( name: 'DEPLOY_ENV', choices: ['dev', 'staging', 'production'], description: 'Which environment to deploy to?' ) booleanParam( name: 'RUN_TESTS', defaultValue: true, description: 'Run tests before deploying?' ) } stages { stage('Deploy') { steps { echo "Deploying to: ${params.DEPLOY_ENV}" sh "kubectl apply -f k8s/${params.DEPLOY_ENV}/" } } } }

When Conditions — Conditional Stage Execution

Jenkinsfile — When Conditions
stage('Deploy to Production') { when { branch 'main' // only deploy when pushing to main branch } steps { sh 'kubectl apply -f k8s/production/' } } stage('Deploy to Staging') { when { branch 'develop' } steps { sh 'kubectl apply -f k8s/staging/' } }

Using Docker as a Jenkins Agent

Instead of installing tools (Java, Node, Python) on your Jenkins server, run each stage inside a Docker container with exactly the tools you need. Clean, isolated, reproducible.

Jenkinsfile — Docker Agent
pipeline { agent none // no default agent — each stage defines its own stages { stage('Build Java App') { agent { docker { image 'maven:3.9-eclipse-temurin-17' } } steps { sh 'mvn clean package' // Maven available here, not on other stages } } stage('Build Node Frontend') { agent { docker { image 'node:18-alpine' } } steps { sh 'npm install && npm run build' } } } }

Managing Credentials Securely

Never hardcode passwords, tokens, or keys in your Jenkinsfile. Jenkins has a built-in Credentials Store — store secrets there and reference them by ID.

Terminal — Add credential via CLI
# Or go to: Jenkins → Manage Jenkins → Credentials → Add Credential # Types available: Username/Password, Secret Text, SSH Key, Certificate
Jenkinsfile — Using Credentials
// Secret text (API token) withCredentials([string(credentialsId: 'aws-token', variable: 'AWS_TOKEN')]) { sh 'aws configure set aws_access_key_id $AWS_TOKEN' } // SSH key for server access withCredentials([sshUserPrivateKey( credentialsId: 'prod-server-ssh', keyFileVariable: 'SSH_KEY' )]) { sh 'ssh -i $SSH_KEY ubuntu@prod-server.com "docker pull myapp:latest"' }
⚠️ Common Mistake

Never echo or print credential variables in pipeline logs. Jenkins masks them automatically — but if you do string interpolation inside sh with double quotes, the value gets exposed in the log. Always use single quotes or withCredentials blocks carefully.

Shared Libraries — DRY Pipelines

If you have 20 microservices each with their own Jenkinsfile, maintaining them all is painful. Shared Libraries let you write pipeline logic once and reuse it across all projects.

GitHub Repo: jenkins-shared-library/ └── vars/ ├── buildAndPush.groovy ← reusable step └── deployToK8s.groovy ← reusable step Any Jenkinsfile can then call: @Library('my-shared-lib') _ buildAndPush(image: 'myapp', tag: BUILD_NUMBER) deployToK8s(env: 'production')

Setting Up Jenkins on EC2

Terminal — Install Jenkins on Ubuntu
# Install Java (Jenkins requires JDK 17+) sudo apt update && sudo apt install fontconfig openjdk-17-jre -y # Add Jenkins repo and install sudo wget -O /usr/share/keyrings/jenkins-keyring.asc \ https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key echo "deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc]" \ https://pkg.jenkins.io/debian-stable binary/ | sudo tee \ /etc/apt/sources.list.d/jenkins.list > /dev/null sudo apt update && sudo apt install jenkins -y # Start Jenkins sudo systemctl enable jenkins && sudo systemctl start jenkins # Get initial admin password sudo cat /var/lib/jenkins/secrets/initialAdminPassword # Access Jenkins at http://YOUR-EC2-IP:8080
🚀 Mini Project — Full CI/CD Pipeline

Build a complete pipeline from push to production:

  1. Install Jenkins on an EC2 instance, install plugins: Git, Docker Pipeline, Kubernetes CLI
  2. Store Docker Hub and GitHub credentials in Jenkins Credentials Store
  3. Create a Jenkinsfile with stages: Checkout → Build (Maven) → Parallel (Unit Tests + Security Scan) → Docker Build & Push → Deploy to K8s
  4. Configure a GitHub Webhook to trigger Jenkins on every push to main
  5. Add a when { branch 'main' } condition so deploys only happen from main
  6. Push a code change — watch the pipeline trigger automatically end to end
  7. Break a test intentionally — verify the pipeline stops before deploying
✅ Module Summary
  • CI = auto-build and test on every push; CD = auto-deploy when tests pass
  • Jenkins uses a Controller + Agent model — never run builds on the Controller
  • Jenkinsfile defines your pipeline as code — version it in Git alongside your app
  • Use parallel stages to run independent checks simultaneously
  • Store all secrets in Jenkins Credentials Store — never hardcode in Jenkinsfile
  • Use Docker as agents to avoid polluting the Jenkins server with tool installations
  • Shared Libraries let you reuse pipeline logic across multiple projects
