Module 04 — Cloud Foundation

AWS Fundamentals

What is AWS?

Amazon Web Services (AWS) is the world's most popular cloud platform — used by Netflix, Airbnb, NASA, and millions of startups and enterprises. Instead of buying and maintaining your own physical servers, you rent computing power, storage, and networking from Amazon and pay only for what you use.

AWS has 200+ services. As a DevOps engineer you don't need to know all of them. This module covers the 6 core services every DevOps engineer works with daily.

🏙️ Analogy — Renting vs Building

Building your own data centre is like buying land, constructing a building, installing generators, cooling systems, security guards, and network cables — before a single line of code runs. AWS is like moving into a serviced office — power, cooling, security, and networking are already handled. You just bring your laptop and start working.

AWS Global Infrastructure

Understanding how AWS is structured geographically is important for building highly available systems.

🌍
Regions
Physical locations around the world (e.g. ap-south-1 = Mumbai, us-east-1 = Virginia). Each region is completely independent. Choose a region close to your users for lower latency.
🏢
Availability Zones (AZs)
Each region has 2–6 AZs. An AZ is one or more data centres with independent power, cooling, and networking. Deploy across multiple AZs for high availability.
🌐
Edge Locations
Hundreds of smaller locations used by CloudFront (CDN) to cache and deliver content close to end users. Reduces latency globally.
🆓
Free Tier
AWS gives new accounts 12 months of free usage on many services — t2.micro EC2, 5GB S3, 750hrs RDS. Perfect for learning and practice.

Service 1 — IAM (Identity & Access Management)

IAM is the first thing you set up in any AWS account. It controls who can do what on which resources. Never use the root account for day-to-day work.

🔑 Analogy — Office ID Cards

An office has different access cards: the security guard can enter the lobby, the developer can access the server room, the manager can access everything. IAM works the same way — each person or service gets exactly the permissions they need, nothing more.

👤
Users
A person or application. Has credentials (password or access keys). Always give users only the permissions they need — principle of least privilege.
👥
Groups
Collection of users. Attach policies to groups, not individual users — much easier to manage. e.g. Developers group, DevOps group, ReadOnly group.
📋
Policies
JSON documents defining what actions are allowed or denied on which resources. AWS provides managed policies. You can also write custom policies.
🎭
Roles
Like a user but for AWS services — e.g. an EC2 instance that needs to access S3 assumes an IAM Role. No permanent credentials needed.
IAM Policy — Allow S3 Read Only
{ "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": [ "s3:GetObject", "s3:ListBucket" ], "Resource": [ "arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*" ] } ] }
⚠️ IAM Best Practices

Never use root account credentials in code or CLI. Never share access keys. Always enable MFA on root and admin accounts. Use IAM Roles for EC2/Lambda instead of embedding access keys in your code.

Service 2 — EC2 (Elastic Compute Cloud)

EC2 is simply a virtual server in the cloud. You choose the OS, CPU, RAM, storage, and networking. You can launch one in under a minute and connect to it over SSH.

📦
Instance Types
t2.micro (free tier, 1vCPU, 1GB RAM) → t3.medium → c5.xlarge (compute optimised) → r5.2xlarge (memory optimised). Choose based on your workload.
💿
AMI
Amazon Machine Image — a snapshot/template of an OS. Use official AWS AMIs (Amazon Linux, Ubuntu) or create your own with pre-installed software.
🔑
Key Pairs
SSH key pair for secure access. AWS stores the public key, you keep the private key (.pem file). Never lose it — you can't regenerate it.
💾
EBS Volumes
Elastic Block Storage — persistent disk for EC2. Like a hard drive attached to your server. Data survives instance stop/restart but not termination (unless set to retain).
Terminal — Launch & Connect to EC2
# Launch a free-tier EC2 (Ubuntu) aws ec2 run-instances \ --image-id ami-0c55b159cbfafe1f0 \ --instance-type t2.micro \ --key-name my-keypair \ --security-group-ids sg-12345678 \ --subnet-id subnet-12345678 # Get the public IP aws ec2 describe-instances \ --query 'Reservations[*].Instances[*].PublicIpAddress' \ --output text # SSH into the server ssh -i "my-keypair.pem" ubuntu@YOUR-PUBLIC-IP # You're now inside a cloud server! # Stop / Start / Terminate aws ec2 stop-instances --instance-ids i-1234567890abcdef0 aws ec2 start-instances --instance-ids i-1234567890abcdef0 aws ec2 terminate-instances --instance-ids i-1234567890abcdef0

Service 3 — VPC (Virtual Private Cloud)

A VPC is your own isolated private network inside AWS. Every EC2 instance, RDS database, and Lambda function lives inside a VPC. You control the IP ranges, subnets, routing, and security.

🏘️ Analogy — Gated Community

A VPC is like a gated community. You define the roads (subnets), the gate rules (security groups), and who can enter or leave (route tables, internet gateway). Your database lives in the private area (no internet access), your web server lives in the public area (accessible to users).

YOUR VPC (10.0.0.0/16) ┌────────────────────────────────────────────────────┐ │ │ │ PUBLIC SUBNET (10.0.1.0/24) │ │ ┌─────────────────────────────┐ │ │ │ Web Server (EC2) │ ← Internet Access │ │ │ Load Balancer │ │ │ └─────────────────────────────┘ │ │ ↕ (internal only) │ │ PRIVATE SUBNET (10.0.2.0/24) │ │ ┌─────────────────────────────┐ │ │ │ Database (RDS) │ ← No Internet │ │ │ App Servers │ │ │ └─────────────────────────────┘ │ │ │ └──────────────────── [Internet Gateway] ────────────┘ ↕ INTERNET
🌐
Internet Gateway
Connects your VPC to the internet. Attach to a VPC and add a route in your route table to enable public subnet internet access.
🗺️
Route Tables
Control where network traffic goes. Public subnet route table has a route to the Internet Gateway. Private subnet route table has no internet route.
🛡️
Security Groups
Virtual firewall for EC2 instances. Stateful — allow inbound rules and outbound is automatically allowed. Define which ports/protocols are permitted.
🚧
NACLs
Network Access Control Lists — firewall at the subnet level. Stateless — you must define both inbound AND outbound rules. Adds a second layer of security.

Service 4 — S3 (Simple Storage Service)

S3 is object storage — an infinite hard drive in the cloud. Store files, images, videos, backups, logs, static websites — anything. Files live in buckets and are accessed via URLs. Bucket names must be globally unique.

Terminal — S3 Operations
# Create a bucket aws s3 mb s3://my-devops-bucket --region ap-south-1 # Upload a file aws s3 cp index.html s3://my-devops-bucket/ # Sync entire folder (like rsync) aws s3 sync ./website s3://my-devops-bucket/ # Download a file aws s3 cp s3://my-devops-bucket/index.html ./ # List bucket contents aws s3 ls s3://my-devops-bucket # Enable static website hosting aws s3 website s3://my-devops-bucket \ --index-document index.html \ --error-document error.html # Delete an object aws s3 rm s3://my-devops-bucket/old-file.txt

Service 5 — CloudWatch (Monitoring & Alerting)

CloudWatch is AWS's built-in monitoring service. Every AWS service automatically publishes metrics to CloudWatch. You can set alarms, create dashboards, and trigger automated actions based on thresholds.

📊
Metrics
EC2 publishes CPU, network, disk metrics every minute. RDS publishes connections, storage, query time. You can also push custom metrics from your application.
🔔
Alarms
Set thresholds and get notified. States: OK, ALARM, INSUFFICIENT_DATA. Actions: send SNS notification, trigger Auto Scaling, stop/reboot an instance.
📋
Logs
Centralise all your application and system logs. Lambda, EC2, RDS, and most AWS services can stream logs directly to CloudWatch Logs.
⚡
EventBridge
React to events in your AWS environment automatically. E.g. EC2 instance state changes → trigger Lambda → send Slack notification.
Terminal — Create a CloudWatch Alarm
# Alert when EC2 CPU > 80% for 2 consecutive 5-minute periods aws cloudwatch put-metric-alarm \ --alarm-name "high-cpu" \ --alarm-description "Alert when CPU exceeds 80%" \ --metric-name CPUUtilization \ --namespace AWS/EC2 \ --statistic Average \ --period 300 \ --evaluation-periods 2 \ --threshold 80 \ --comparison-operator GreaterThanThreshold \ --dimensions Name=InstanceId,Value=i-1234567890abcdef0 \ --alarm-actions arn:aws:sns:ap-south-1:123456:my-alerts

Service 6 — AWS CLI (Command the Cloud)

The AWS CLI lets you interact with every AWS service from the terminal — no browser needed. Essential for automation, scripting, and CI/CD pipelines.

Terminal — AWS CLI Setup
# Install on Ubuntu curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip" unzip awscliv2.zip && sudo ./aws/install # Configure with IAM credentials aws configure AWS Access Key ID: AKIAIOSFODNN7EXAMPLE AWS Secret Access Key: wJalrXUtnFEMI/K7MDENG... Default region name: ap-south-1 Default output format: json # Verify — shows which account you're connected to aws sts get-caller-identity { "Account": "123456789012", "UserId": "AIDAIOSFODNN7EXAMPLE" } # Use profiles for multiple AWS accounts aws configure --profile production aws s3 ls --profile production
💡 Useful CLI Tricks

Use --output table for readable output, --query to filter JSON results, and --dry-run to test commands without executing them. For scripts, --output text is easiest to parse.

✅ Module Summary
  • AWS regions = physical locations; AZs = isolated data centres within a region
  • IAM — always use least privilege; use Roles for services, never embed access keys in code
  • EC2 — virtual servers; choose instance type based on workload; access via SSH + key pair
  • VPC — your private network; public subnets for web servers, private subnets for databases
  • S3 — unlimited object storage; use for static websites, backups, logs, and artefacts
  • CloudWatch — metrics, alarms, logs for all AWS resources; always set CPU and cost alarms
  • AWS CLI — essential for automation; configure profiles for multiple accounts
