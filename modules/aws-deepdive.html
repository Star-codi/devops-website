AWS Deep Dive — Production Services

Beyond the Basics

You know EC2, S3, IAM, VPC, and CloudWatch from AWS Fundamentals. This module goes deeper into the services you'll use when building production-grade architectures — managed databases, serverless functions, load balancing, auto scaling, S3 storage optimisation, and more.

RDS — Managed Relational Databases

Amazon RDS (Relational Database Service) manages your database for you — patching, backups, replication, and failover are all automated. You just connect and query. Supported engines: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Amazon Aurora.

🍽️ Analogy — Restaurant vs Home Cooking

Self-managing a database on EC2 is like cooking at home — you buy ingredients, cook, clean up, and handle everything yourself. RDS is like eating at a restaurant — you just order what you need. The kitchen (patching, backups, replication) is handled for you.

🔄
Multi-AZ
RDS automatically maintains a standby replica in a different AZ. If the primary fails, RDS automatically fails over to the standby in ~60 seconds. Zero data loss.
📖
Read Replicas
Offload read traffic to one or more read replicas. Improves performance for read-heavy workloads. Can be in different regions for global read performance.
💾
Automated Backups
RDS takes daily snapshots and transaction logs. You can restore to any point in time within your retention window (up to 35 days).
⚡
Aurora
AWS's own MySQL/PostgreSQL-compatible engine. Up to 5x faster than standard MySQL. Storage auto-scales to 128TB. Serverless option available.
Terminal — Create RDS Instance
# Create a PostgreSQL RDS instance aws rds create-db-instance \ --db-instance-identifier my-postgres-db \ --db-instance-class db.t3.micro \ --engine postgres \ --master-username admin \ --master-user-password MySecurePass123 \ --allocated-storage 20 \ --multi-az \ --no-publicly-accessible \ --db-subnet-group-name my-db-subnet-group # Get the endpoint to connect aws rds describe-db-instances \ --query 'DBInstances[*].Endpoint.Address' "my-postgres-db.abc123.ap-south-1.rds.amazonaws.com" # Connect with psql psql -h my-postgres-db.abc123.ap-south-1.rds.amazonaws.com \ -U admin -d postgres
⚠️ Always put RDS in a Private Subnet

Never set --publicly-accessible true on a production database. Place RDS in a private subnet — only EC2 instances and Lambda functions inside the same VPC should be able to reach it. Use a security group that only allows connections from your application's security group.

DynamoDB — Serverless NoSQL Database

DynamoDB is a fully managed NoSQL database. No servers to manage, no connections to configure. It scales automatically to handle millions of requests per second with single-digit millisecond latency.

SchemaFixed — define columns upfrontFlexible — each item can differ

LatencyMillisecondsSingle-digit milliseconds
QueriesComplex SQL, JOINs, transactionsSimple key-based, limited queries
Best forFinancial data, complex relationshipsReal-time apps, gaming, IoT, caching
Terminal — DynamoDB Operations
# Create a table aws dynamodb create-table \ --table-name Users \ --attribute-definitions AttributeName=userId,AttributeType=S \ --key-schema AttributeName=userId,KeyType=HASH \ --billing-mode PAY_PER_REQUEST # Put an item aws dynamodb put-item \ --table-name Users \ --item '{"userId": {"S": "u001"}, "name": {"S": "Arjun"}, "role": {"S": "DevOps"}}' # Get an item by key aws dynamodb get-item \ --table-name Users \ --key '{"userId": {"S": "u001"}}'

Lambda — Serverless Functions

Lambda lets you run code without provisioning or managing any servers. Upload your function, define a trigger, and AWS handles everything. You pay only per invocation and execution duration — nothing when idle.

💡 Analogy — Pay-Per-Use Electricity

Running a server 24/7 is like buying a generator and running it constantly — even at 3am when nobody is using it. Lambda is like the electrical grid — you only pay for the exact milliseconds your code actually runs. Zero cost when idle.

Lambda Function — Python (Resize S3 Image)
import boto3 import json s3 = boto3.client('s3') def lambda_handler(event, context): # Triggered when a file is uploaded to S3 bucket = event['Records'][0]['s3']['bucket']['name'] key = event['Records'][0]['s3']['object']['key'] print(f"New file uploaded: {key} in {bucket}") # Download, process, re-upload logic here return { 'statusCode': 200, 'body': json.dumps('Processed successfully') }
Terminal — Deploy Lambda
# Zip your function code zip function.zip lambda_function.py # Create Lambda function aws lambda create-function \ --function-name my-image-processor \ --runtime python3.11 \ --role arn:aws:iam::123456789012:role/lambda-role \ --handler lambda_function.lambda_handler \ --zip-file fileb://function.zip \ --timeout 30 \ --memory-size 256 # Invoke manually for testing aws lambda invoke \ --function-name my-image-processor \ --payload '{"test": "event"}' \ response.json # View Lambda logs in CloudWatch aws logs tail /aws/lambda/my-image-processor --follow

ELB & Auto Scaling — High Availability at Scale

A single EC2 instance will eventually fail or get overwhelmed. Elastic Load Balancer (ELB) distributes traffic across multiple instances. Auto Scaling automatically adds or removes instances based on demand.

USERS ↓ [Application Load Balancer] ← distributes traffic ↓ ↓ ↓ [EC2-1] [EC2-2] [EC2-3] ← Auto Scaling Group (scales 2–20 instances) Auto Scaling watches CloudWatch metrics: CPU > 70% → add instances CPU < 30% → remove instances
⚖️
Application LB (ALB)
Layer 7 (HTTP/HTTPS). Routes based on URL path, hostname, or headers. Best for web apps and microservices. Supports WebSockets and HTTP/2.
🔌
Network LB (NLB)
Layer 4 (TCP/UDP). Extremely low latency, millions of requests/sec. Use for high-performance, non-HTTP traffic — databases, gaming, IoT.
📈
Auto Scaling Groups
Define min, max, and desired instance count. Scaling policies trigger based on CloudWatch alarms. New instances launch from a Launch Template.
🎯
Target Tracking
Simplest scaling policy — set a target metric (e.g. 60% CPU) and Auto Scaling maintains it automatically. No need to define scale-up/scale-down thresholds separately.

SNS & SQS — Messaging Services

📢
SNS — Simple Notification Service
Pub/Sub messaging. Publish one message to a Topic and it fans out to all subscribers (email, SMS, Lambda, SQS, HTTP). Use for alerts, broadcasts, and triggering multiple services from one event.
📬
SQS — Simple Queue Service
Message queue. Producer puts messages in, consumer reads them. Decouples services — producer doesn't wait for consumer. Messages stored until processed (up to 14 days).
SNS (Fan-out pattern): S3 Upload → SNS Topic → Email notification → Lambda (resize image) → SQS (process later) SQS (Decoupling pattern): Web App → SQS Queue → Lambda Worker → Database (producer doesn't wait — queue buffers the load)
🚀 Mini Project — Serverless Image Processing Pipeline
  1. Create two S3 buckets: uploads-bucket and processed-bucket
  2. Create a Lambda function (Python) triggered on S3 uploads — it logs the filename and copies it to processed-bucket
  3. Create an SNS topic and subscribe your email to it
  4. Update the Lambda to publish a message to SNS after processing
  5. Set up a Lifecycle Policy on uploads-bucket — move objects to Standard-IA after 30 days
  6. Add a CloudWatch alarm that triggers if Lambda errors > 0 and sends an SNS alert
  7. Upload a test file — verify email notification arrives and file appears in processed bucket
✅ Module Summary
  • RDS = managed relational databases; always use Multi-AZ + private subnet in production
  • DynamoDB = serverless NoSQL; use for high-throughput, flexible schema workloads
  • Lambda = serverless functions; pay per invocation, scales automatically, zero idle cost
  • S3 storage classes — Standard for frequent access, Glacier for archives; use Lifecycle Policies to automate transitions and cut costs
  • ELB distributes traffic across instances; ALB for HTTP apps, NLB for TCP/high-performance
  • Auto Scaling adjusts instance count based on demand — set min/max/desired carefully
  • SNS for fan-out notifications; SQS for decoupling services with message queues
