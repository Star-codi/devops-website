Module 13 — Configuration Management

Ansible — Automate Everything

What Problem Does Ansible Solve?

Terraform just created 10 EC2 servers for you. Now what? Someone has to install Nginx, configure firewalls, create users, deploy your app, and apply security patches — on all 10 servers. Doing this manually by SSH-ing into each one is slow, error-prone, and inconsistent.

Ansible automates all of this. You describe the desired state of your servers in simple YAML files called Playbooks, and Ansible connects to all your servers over SSH and makes it happen — simultaneously, consistently, on any number of machines.

👨‍🍳 Analogy — Recipe for a Chef

Imagine you're opening 10 restaurant branches. You need every kitchen set up identically — same equipment, same layout, same processes. You could personally visit each branch and set it up manually. Or you could write a detailed recipe/manual and hand it to a team that sets up all 10 branches simultaneously from the same instructions. Ansible is that recipe — and it executes itself.

How Ansible Works — No Agent Required

Unlike most tools, Ansible is agentless. You don't install anything on the servers you're managing. Ansible connects via plain SSH, runs tasks, and disconnects. That's it.

┌─────────────────────────────────────────────────────┐ │ YOUR MACHINE (Control Node) │ │ │ │ ansible-playbook site.yml │ │ │ │ │ reads Inventory (list of servers) │ │ reads Playbook (list of tasks) │ └───────┬──────────────────────────────────────────────┘ │ SSH (port 22) — no agent needed on servers ├──────────────────┬──────────────────┐ ▼ ▼ ▼ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Server 1 │ │ Server 2 │ │ Server 3 │ │ (web-01) │ │ (web-02) │ │ (db-01) │ │ 10.0.1.10 │ │ 10.0.1.11 │ │ 10.0.2.10 │ └──────────────┘ └──────────────┘ └──────────────┘ Tasks run in parallel across all servers simultaneously

Installing Ansible

Terminal — Install Ansible
# Ubuntu / Debian sudo apt update && sudo apt install ansible -y # macOS brew install ansible # pip (any OS) pip install ansible # Verify installation ansible --version ansible [core 2.16.0] python version = 3.11.0

Inventory — Tell Ansible About Your Servers

The inventory is a list of all the servers Ansible should manage. You can group servers by role — web servers, database servers, etc. — and target specific groups in your playbooks.

inventory.ini — Static Inventory
# Individual server [webservers] web-01 ansible_host=10.0.1.10 web-02 ansible_host=10.0.1.11 web-03 ansible_host=10.0.1.12 [databases] db-01 ansible_host=10.0.2.10 db-02 ansible_host=10.0.2.11 # Group of groups [production:children] webservers databases # Variables for a group [webservers:vars] ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/my-key.pem ansible_python_interpreter=/usr/bin/python3
Terminal — Test Connectivity
# Ping all servers to verify SSH works ansible all -i inventory.ini -m ping web-01 | SUCCESS => { "ping": "pong" } web-02 | SUCCESS => { "ping": "pong" } db-01 | SUCCESS => { "ping": "pong" } # Run an ad-hoc command on all webservers ansible webservers -i inventory.ini -m command -a "uptime" web-01 | SUCCESS | rc=0 >> 14:23:01 up 2 days, 3:12 web-02 | SUCCESS | rc=0 >> 14:23:01 up 2 days, 3:14 # Check disk space on all servers ansible all -i inventory.ini -a "df -h"

Playbooks — The Heart of Ansible

A Playbook is a YAML file that describes a series of tasks to run on your servers. Each task uses an Ansible module — a pre-built unit of work (install a package, copy a file, start a service, create a user, etc.).

webserver.yml — Install & Configure Nginx
--- - name: Configure web servers hosts: webservers # target the webservers group become: true # use sudo for all tasks tasks: - name: Update apt cache apt: update_cache: yes cache_valid_time: 3600 # only update if cache is older than 1hr - name: Install Nginx apt: name: nginx state: present # present = installed, absent = removed - name: Copy custom Nginx config template: src: templates/nginx.conf.j2 # Jinja2 template on your machine dest: /etc/nginx/nginx.conf owner: root group: root mode: '0644' notify: Restart Nginx # trigger handler if this changes - name: Ensure Nginx is started and enabled service: name: nginx state: started enabled: yes # start on boot - name: Open port 80 in UFW firewall ufw: rule: allow port: '80' proto: tcp handlers: # Handlers run ONLY when notified, and only once at the end - name: Restart Nginx service: name: nginx state: restarted
Terminal — Run a Playbook
# Run the playbook ansible-playbook -i inventory.ini webserver.yml PLAY [Configure web servers] ******************************************* TASK [Install Nginx] *************************************************** changed: [web-01] changed: [web-02] TASK [Ensure Nginx is started] ***************************************** ok: [web-01] ok: [web-02] PLAY RECAP ************************************************************* web-01 : ok=4 changed=2 unreachable=0 failed=0 web-02 : ok=4 changed=2 unreachable=0 failed=0 # Dry run — see what WOULD change without applying ansible-playbook -i inventory.ini webserver.yml --check # Run with verbose output for debugging ansible-playbook -i inventory.ini webserver.yml -v # Run only specific tasks using tags ansible-playbook -i inventory.ini webserver.yml --tags "nginx" # Target a specific server only ansible-playbook -i inventory.ini webserver.yml --limit web-01
💡 Idempotency — Run It 10 Times, Same Result

Ansible tasks are idempotent — running a playbook 10 times produces the same result as running it once. If Nginx is already installed, the "Install Nginx" task shows ok (no change) instead of changed. This means you can safely re-run playbooks to fix drift or apply updates without breaking anything.

Variables — Make Playbooks Flexible

vars.yml — Variables File
--- app_name: my-webapp app_port: 3000 app_user: appuser app_dir: /opt/my-webapp packages: - nginx - nodejs - git
playbook.yml — Using Variables
--- - name: Deploy application hosts: webservers become: true vars_files: - vars.yml tasks: - name: Install required packages apt: name: "{{ item }}" # loop over packages list state: present loop: "{{ packages }}" - name: Create app user user: name: "{{ app_user }}" shell: /bin/bash state: present - name: Create app directory file: path: "{{ app_dir }}" state: directory owner: "{{ app_user }}" mode: '0755'

Ansible Vault — Encrypting Secrets

Never store passwords or API keys in plain text in your playbooks or variable files. Ansible Vault encrypts sensitive files so they can be safely committed to Git.

Terminal — Ansible Vault
# Create an encrypted variables file ansible-vault create secrets.yml # Opens editor — type your secrets: # db_password: supersecret # api_key: abc123xyz # Encrypt an existing file ansible-vault encrypt vars/secrets.yml Encryption successful # View encrypted file ansible-vault view secrets.yml # Edit encrypted file ansible-vault edit secrets.yml # Run playbook with vault password ansible-playbook site.yml --ask-vault-pass # Or use a password file (for CI/CD) ansible-playbook site.yml --vault-password-file ~/.vault_pass

Roles — Organise Playbooks at Scale

As your infrastructure grows, a single playbook file becomes hard to maintain. Roles let you split your playbook into reusable, well-structured components — one role per responsibility (nginx, docker, users, monitoring).

roles/ ├── nginx/ │ ├── tasks/ │ │ └── main.yml ← all nginx tasks │ ├── handlers/ │ │ └── main.yml ← restart nginx handler │ ├── templates/ │ │ └── nginx.conf.j2 ← Jinja2 config template │ ├── vars/ │ │ └── main.yml ← nginx-specific variables │ └── defaults/ │ └── main.yml ← default variable values ├── docker/ │ └── tasks/main.yml └── users/ └── tasks/main.yml
site.yml — Using Roles
--- - name: Configure web servers hosts: webservers become: true roles: - users # create system users first - nginx # install and configure nginx - docker # install docker - name: Configure database servers hosts: databases become: true roles: - users - postgresql
Terminal — Create a Role
# Ansible generates the folder structure automatically ansible-galaxy init roles/nginx - Role roles/nginx was created successfully # Install a community role from Ansible Galaxy ansible-galaxy install geerlingguy.docker # requirements.yml — pin your role versions # roles: # - name: geerlingguy.docker # version: 6.1.0 ansible-galaxy install -r requirements.yml

Jinja2 Templates — Dynamic Config Files

Templates let you generate configuration files dynamically using variables. The same template produces different configs for dev, staging, and production.

templates/nginx.conf.j2 — Jinja2 Template
server { listen 80; server_name {{ server_name }}; {# variable from vars file #} location / { proxy_pass http://127.0.0.1:{{ app_port }}; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; } {% if enable_ssl %} {# conditional block #} listen 443 ssl; ssl_certificate /etc/ssl/certs/cert.pem; ssl_certificate_key /etc/ssl/private/key.pem; {% endif %} {% for upstream in upstream_servers %} {# loop #} upstream backend { server {{ upstream }}; } {% endfor %} }

Terraform + Ansible Together — The Full Flow

This is how real DevOps teams provision and configure infrastructure end to end:

Step 1 — Terraform provisions infrastructure terraform apply → Creates EC2 instances, VPC, security groups → Outputs IP addresses Step 2 — Generate dynamic Ansible inventory from Terraform output terraform output -json | python3 generate_inventory.py > inventory.ini Step 3 — Ansible configures the servers ansible-playbook -i inventory.ini site.yml → Installs packages, configures Nginx, deploys app, sets up monitoring Result: Fully provisioned + configured infrastructure from code
Makefile — Full Workflow
provision: terraform apply -auto-approve terraform output -json > tf_output.json python3 scripts/generate_inventory.py configure: ansible-playbook -i inventory.ini site.yml deploy: provision configure @echo "Infrastructure provisioned and configured" destroy: terraform destroy -auto-approve
🚀 Mini Project — Provision with Terraform, Configure with Ansible
  1. Use Terraform to create 2 EC2 instances (web servers) and output their IPs
  2. Write an inventory.ini using the output IPs
  3. Create an Ansible Role called webserver with tasks to: update packages, install Nginx and Node.js, create an app user, copy a config template, start and enable Nginx
  4. Create a secrets.yml with Ansible Vault containing a fake DB password — reference it in your playbook
  5. Write site.yml that applies the webserver role to the webservers group
  6. Run the playbook — verify Nginx is accessible on both server IPs
  7. Change a variable (e.g. app port) and re-run — verify idempotency (only changed tasks show "changed")
  8. Clean up with terraform destroy
✅ Module Summary
  • Ansible configures servers — Terraform creates them. Use both together.
  • Agentless — connects via SSH, nothing to install on target servers
  • Inventory defines which servers to manage, grouped by role
  • Playbooks are YAML files describing tasks to run — readable by anyone
  • Tasks are idempotent — safe to re-run without breaking anything
  • Handlers only run when notified and only once — perfect for service restarts
  • Use Ansible Vault to encrypt secrets — never commit plain text passwords
  • Roles organise large playbooks into reusable, shareable components
  • Jinja2 templates generate dynamic config files from variables
